# プロジェクト指示文

## 基本ルール

### 1. 実装前の確認プロセス

- 実装に入る前に、理解した仕様を箇条書きで整理して提示すること。
- 不明点や曖昧な点があれば、推測で進めず必ず質問すること。
- 仕様の確認が取れてから実装を開始すること。

### 2. 影響範囲の事前調査

- 機能の追加・変更を行う際は、実装前に影響を受ける既存の関数・シート・トリガーを洗い出し、リストとして提示すること。
- 横展開が必要な箇所がある場合は、対応漏れがないよう明示すること。
- 影響範囲の確認が取れてから実装を開始すること。

### 3. 段階的な進め方

- 大きな機能は一度に実装せず、以下のような段階に分けて進めること。
  1. データ構造・設計方針の提示 → 確認
  2. 主要ロジックの実装 → 確認
  3. スプレッドシートとの連携 → 確認
  4. 結合・仕上げ → 確認
- 各段階でユーザーの確認を取ってから次に進むこと。

### 4. その他

- 問題報告時は、まず原因を説明し、修正不要な場合はその旨を先に伝える。修正が必要な場合のみ対応方針を提示する。すぐに修正に飛びつかない。

## GAS実装上の注意点

### スプレッドシートへのアクセス

- シート名は定数（`SHEET_NAME_RESPONSES` / `SHEET_NAME_WEATHER_LOG` / `SHEET_NAME_LOG`）で管理されている。
- `getDataRange().getValues()` は全データを取得するため、大量行になる場合はパフォーマンスに注意すること。

### トリガー管理

- GASの時間トリガーには±15分の誤差があるため、厳密な時刻制御には向かない点を考慮すること。
- トリガー変更を伴う実装の場合は、変更内容を事前に明示すること。

### 外部API呼び出し

- `UrlFetchApp.fetch()` は同期処理だが、GASの実行時間制限（6分）に注意すること。
- スプレッドシートID・緯度経度はスクリプトプロパティから取得する。コード中にハードコードしないこと。

## API構成

| 機能 | エンドポイント | 認証 |
|---|---|---|
| AMeDAS観測点テーブル | `https://www.jma.go.jp/bosai/amedas/const/amedastable.json` | 不要 |
| AMeDAS時系列データ | `https://www.jma.go.jp/bosai/amedas/data/point/{観測点コード}/{YYYYMMDD}_{HH}.json` | 不要 |
| カレンダー（祝日判定） | GAS組み込み `CalendarApp` | OAuth（自動） |
| スプレッドシート | GAS組み込み `SpreadsheetApp` | OAuth（自動） |

### AMeDAS API利用上の注意点

- **3時間ごとのバッチ配信**: `{HH}` は `00, 03, 06, 09, 12, 15, 18, 21` のいずれか。1ファイルに3時間分（10分ごと）のデータが含まれる。
- **観測点別データの保持期間は約10日**。それより古い時点のデータは取得不可。
- **気圧・湿度を観測している観測点は限定的**（全観測点約1,300のうち主要約150観測点のみ）。観測点選定時は `elems` フィールド（14桁の観測項目フラグ）の3桁目（気温）・4桁目（湿度）・10桁目（気圧）が `1` であることを確認すること。
- 非公式APIのため**仕様変更リスクあり**。エラー時はAPI仕様変更を疑うこと。

### スクリプトプロパティ一覧

| キー名 | 説明 |
|---|---|
| `SPREADSHEET_ID` | GoogleスプレッドシートのID |
| `LAT` | 緯度（例: `35.6895`） |
| `LON` | 経度（例: `139.6917`） |
| `AMEDAS_STATION_CODE` | AMeDAS観測点コード。未設定時は `LAT`/`LON` から最寄り観測点を自動選定して保存する。 |
